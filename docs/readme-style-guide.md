# MCGR README Style Guide

This guide keeps README wording consistent across the MCGR repository family.

## Tone

- Executive-friendly in hub repositories
- Practical and implementation-oriented in spoke repositories
- Clear, direct, and reusable
- Avoid overly formal or repetitive language

## Preferred Structure

Use the same sequence in every README:

1. Overview
2. Why It Matters
3. Core Model
4. Key Artifacts
5. Implementation Guidance
6. Examples
7. References
8. Start Here

## Wording Rules

- Use short, plain sentences.
- Prefer active voice.
- Explain the value before the detail.
- Link to related repos instead of repeating long explanations.
- Use consistent nouns for the same concepts:
  - hub
  - core framework
  - applied operating repo
  - supporting knowledge repo

## Recommended Phrasing

- "This repository provides..."
- "Use this repo to..."
- "The next step is..."
- "This framework helps teams..."
- "Related repositories"

## Avoid

- Duplicate ecosystem descriptions in every repo
- Overly promotional language
- Long paragraphs that hide the action
- Mixed terms for the same concept

## Example Style

Good:

> This repository provides a practical assessment toolkit for cloud governance, SRE, DR, FinOps, and security controls.

Less clear:

> This repository is a comprehensive and transformative solution intended to empower stakeholders with a wide range of capabilities.

## Cross-Repo Consistency

When updating README files across the ecosystem:

- keep the opening summary aligned to the repo type
- keep the section order stable
- keep the start-here steps short and action-oriented
- keep links to the hub repo visible

## Next Use

Use this guide before adjusting any README text in the MCGR ecosystem so the whole family reads like one connected product set.
